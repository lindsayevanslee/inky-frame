# Belcourt spider — GitHub Actions scheduling delay analysis

Quantifies how late the `run belcourt spider` workflow actually ran versus its
scheduled cron time, which motivated moving scheduling to cron-job.org
(see [.github/workflows/actions.yml](../.github/workflows/actions.yml)).

## Files

| Path | What it is | How to regenerate |
|---|---|---|
| `data/spider_runs_raw.json` | Raw run records — an **append-only archive**. The source of truth. May reach further back than GitHub's live API still retains, because purged runs captured in earlier refreshes are preserved here. | `refresh_data.py --pull` (merge) |
| `data/manifest.json` | Provenance for the archive: when last refreshed, coverage window (earliest/latest run), and counts. Rebuilt on every refresh. | `refresh_data.py` |
| `data/schedule_history.json` | The cron schedule as configured at each commit, with the commit's **UTC** instant. Reconstructed from git history of the workflow file. | regenerated by the snippet below |
| `data/spider_delays.csv` | Derived per-run table: matched fire time, delay (min), midnight-crossing flag, conclusion. | `spider_delay_analysis.py` |
| `spider_delay.png` | The chart. | `spider_delay_analysis.py` |
| `spider_delay_analysis.py` | Computes delays and renders the chart from the two stored JSON files. No live API/git needed. | — |

## Re-running the analysis (uses stored data, no network)

```sh
uv run --with matplotlib python3 analysis/spider_delay_analysis.py
```

## Refreshing the raw data (hits GitHub — do this to add new runs)

**Always use the refresh script — never overwrite `spider_runs_raw.json` with a
raw `gh api` redirect.** The script does a lossless **merge by run `id`** so that
runs GitHub has since purged from its API stay in the archive. A plain
`gh api > file` would silently delete history older than GitHub's retention
window.

```sh
# Pull live from GitHub, merge into the archive, update the manifest:
uv run python3 analysis/refresh_data.py --pull
```

How the merge behaves (verified with a simulated future pull):

- runs only in the archive (purged from the live API) are **kept**;
- runs in both are **updated** from live (so a run's final status/conclusion wins);
- runs only in live are **added**;
- run `id` is unique and immutable, so the merge is idempotent — re-running
  never duplicates or drops.

The schedule history (`schedule_history.json`) is rebuilt from git; re-run the
small Python block in the project chat history (or re-derive: for each commit
touching the workflow, record its committer-date in UTC and the active
uncommented `cron:` line).

## How delay is computed (and why it's correct)

- **UTC throughout.** GitHub returns `run_started_at` in UTC and GitHub cron
  fires on UTC. Delay = actual UTC start − the UTC cron fire instant. No local
  timezone enters the math, so **DST never affects the delay** (it only shifts
  what Central wall-clock a fixed UTC time maps to).
- **Right schedule per run.** Boundaries are instant-based: each run is judged
  against the cron that was active at the run's *instant*, not merely its date.
  Within the retained data window only `0 10 * * *` then `7 10 * * *` applied.
- **Midnight-safe matching.** Each run is matched to the latest *expected fire
  instant ≤ its start*, generated independently from the schedule timeline. A
  run delayed past UTC midnight correctly attributes to the prior day's fire
  (flagged `crossed_utc_midnight`). Verified with a synthetic 15.5h-late case.
- **Anomalies surfaced, not hidden.** The script reports skipped fires (an
  expected fire with no run) and doubled fires (2+ runs for one fire).

## Caveats

- **Coverage = GitHub's retention window** (~2025-04 onward at time of writing),
  not the action's full life since 2023. Earlier `*/5` and `0 13` schedules
  predate any retained runs.
- **Silently-skipped runs are invisible.** If GitHub dropped a scheduled run
  under load, it isn't in the API list, so real reliability is at best as good
  as shown. (As of the last refresh: 0 skipped fires detected in-window.)
